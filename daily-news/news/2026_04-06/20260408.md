#### SaaS連携事業者の侵害後、Snowflake顧客がデータ窃取攻撃の被害に
https://www.bleepingcomputer.com/news/security/snowflake-customers-hit-in-data-theft-attacks-after-saas-integrator-breach/

- 要約
    - 12社超が SaaS 連携事業者の侵害と認証トークン窃取を受けた後、データ窃取攻撃に遭い、その大半が Snowflake 利用企業だった。
    - Snowflake は少数顧客で異常活動を確認し、影響の可能性があるアカウントを直ちにロックダウンして、対象顧客へ通知した。
    - 同社は、自社システムの脆弱性や侵害は否定し、原因は特定の外部連携に結び付いた不正トークン利用だと説明している。
    - 複数の情報源は発端を Anodot のセキュリティ事故とし、ShinyHunters も関与を認め、数十社分のデータ窃取を主張した。
    - 攻撃者は Salesforce からのデータ窃取も試みたが、記事では AI 検知により阻止されたとされ、Google も事案を追跡中だという。
- IOCの列挙

- 推奨事項
    - 推奨事項なし
- その他

- ChatGPTの推奨事項

#### 米国、重要インフラを標的とするイラン系ハッカーに警告
https://www.bleepingcomputer.com/news/security/us-warns-of-iranian-hackers-targeting-critical-infrastructure/

- 要約
    - FBI、CISA、NSA など米政府機関は、インターネット公開中の Rockwell/Allen-Bradley 製 PLC を狙うイラン系攻撃者について合同で警告した。
    - 攻撃は米国の政府施設、水・廃水、エネルギーなど複数の重要インフラ分野に及び、2026年3月以降に金銭的損失と運用妨害を生じさせた。
    - 当局によると、攻撃者は PLC の project file を抜き取り、HMI や SCADA 画面に表示されるデータを改ざんし、混乱や障害を引き起こそうとしている。
    - 近年のイラン系活動は、イランと米国・イスラエル間の敵対行為を背景に活発化した可能性があり、今回の作戦もその流れの一部として説明されている。
    - 防御策として、PLC のインターネット切断または FW 保護、OT への MFA、最新ファームウェア適用、未使用サービスや既定認証の無効化が勧告された。
- IOCの列挙
    - IOC情報なし
- 推奨事項
    - PLC をインターネットから切り離すか、ファイアウォールで保護する。
    - 侵害指標の確認、OT ポート上の不審通信監視、特に海外ホスティング事業者由来の通信確認を行う。
    - OT ネットワークへの MFA 適用、PLC の最新ファームウェア更新、未使用サービスや既定認証鍵の無効化を実施する。
- その他
    - 攻撃者はイラン系 APT とされ、米当局は最近の攻撃活発化がイランと米国・イスラエルの敵対行為への反応である可能性を示している。
- ChatGPTの推奨事項
    - 直ちにインターネット公開中の PLC を棚卸しし、外部公開停止または FW 制限、OT への MFA 適用、既定認証の無効化を実施してください。

#### 最大深刻度の Flowise RCE 脆弱性、攻撃で悪用開始
https://www.bleepingcomputer.com/news/security/max-severity-flowise-rce-vulnerability-now-exploited-in-attacks/

- 要約
    - BleepingComputer は、Flowise の最大深刻度脆弱性 CVE-2025-59528 が実際の攻撃で悪用され始めたと報じている。
    - この欠陥は CustomMCP ノードの mcpServerConfig を安全確認なしに評価する問題で、任意の JavaScript 実行からコマンド実行やファイルアクセスに至る。
    - 脆弱性は 2025 年 9 月に公開済みで、開発元は Flowise 3.0.6 で修正し、現行最新版は 3.1.1 だと記事は説明している。
    - VulnCheck は Canary ネットワークで初の悪用を検知し、現時点では単一の Starlink IP 由来に見える限定的な活動だと述べた。
    - ただし公開中の Flowise インスタンスは 12,000〜15,000 台あるとされ、他の Flowise 脆弱性と合わせて継続的な悪用が懸念されている。
- IOCの列挙
    - IOC情報なし
- 推奨事項
    - Flowise をできるだけ早く 3.1.1 へ更新する。
    - 少なくとも 3.0.6 以上へ更新する。
    - 外部アクセスが不要な場合は、Flowise インスタンスを公開インターネットから外すことを検討する。
- その他
    - 既に VulnCheck の Canary ネットワークで実際の悪用が検知されており、発見や修正情報のみではない。
- ChatGPTの推奨事項
    - Flowise の導入版数と公開状況を直ちに確認し、3[.]0[.]6 未満は緊急更新、不要な外部公開は即時停止してください。

#### 当局、Microsoft 365 のログイン窃取に使われたルーターの DNS 乗っ取りを阻止
https://www.bleepingcomputer.com/news/security/authorities-disrupt-dns-hijacks-used-to-steal-microsoft-365-logins/

- 要約
    - 国際法執行機関と民間企業は、MikroTik や TP-Link ルーターのローカル通信を乗っ取り、Microsoft 資格情報を盗む FrostArmada を妨害した。
    - この作戦は主に SOHO ルーターの DNS 設定を書き換え、攻撃者管理の VPS を DNS リゾルバ化して認証トラフィックを AitM プロキシへ転送していた。
    - 被害者側で見える兆候は主に無効な TLS 証明書警告だけで、これを無視すると Microsoft ログイン情報や OAuth トークンが中間点で収集された。
    - 2025年12月時点で 120 か国 18,000 台に感染が拡大し、政府機関、法執行機関、IT・ホスティング事業者、自前サーバー運用組織が主な標的とされた。
    - Microsoft、Black Lotus Labs、FBI、米司法省、ポーランド政府の連携で攻撃インフラは停止され、記事は証明書ピンニングや公開面の縮小を推奨している。

    - パッチ適用、公開 Web への露出削減、サポート終了機器の撤去によって攻撃面を最小化する。

#### 新たな「GPUBreach」攻撃で、GPU Rowhammer によるシステム乗っ取りが可能に
https://www.bleepingcomputer.com/news/security/new-gpubreach-attack-enables-system-takeover-via-gpu-rowhammer/

- 要約
    - トロント大学の研究者は、GDDR6 を対象に GPU 上で Rowhammer を起こし、GPU ページテーブルを破壊して権限昇格に至る新手法「GPUBreach」を公開した。
    - この攻撃では、権限のない CUDA カーネルが任意の GPU メモリ読み書きを獲得し、さらに NVIDIA ドライバーのメモリ安全性の欠陥と連鎖させて全体侵害へ進み得る。
    - 記事では、IOMMU が有効でも GPUBreach は防げず、従来の DMA 防御だけでは不十分で、root shell 取得にまで至る可能性があると説明している。
    - 研究者らは NVIDIA RTX A6000 で結果を示し、以前の GPUHammer がデータ破壊中心だったのに対し、今回は権限昇格まで可能になった点が強調されている。
    - 研究内容は 2025年11月11日に NVIDIA、Google、AWS、Microsoft に報告済みで、2026年4月13日に技術論文と再現用コードが公開予定とされている。
- IOCの列挙
    - IOC情報なし
- 推奨事項
    - リスクのある利用者は、IOMMU のみを防御策として過信しない。
    - エンタープライズ環境では、Rowhammer 系攻撃を防ぐため System Level Error-Correcting Codes を有効化する。
    - コンシューマ向けで ECC 非対応の GPU については、GPUBreach に対する十分な緩和がない点に留意する。
- その他
    - 本記事は、研究者が発見してベンダーへ報告した新しい攻撃手法と緩和策を扱っており、既に実運用で悪用されているとは記載されていない。
    - 攻撃者の属性情報は記載されておらず、特定の国家や犯罪グループによる実攻撃キャンペーンの説明ではない。
- ChatGPTの推奨事項
    - NVIDIA GDDR6 搭載 GPU の利用環境を直ちに棚卸しし、企業向けでは System Level ECC を有効化し、ECC 非対応 GPU では信頼できない CUDA ワークロードの実行を停止してください。

#### OpenAI Codexのコマンドインジェクション脆弱性がGitHubトークン侵害につながる仕組み
https://www.beyondtrust.com/blog/entry/openai-codex-command-injection-vulnerability-github-token

- 要約
    - BeyondTrust は、OpenAI Codex の branch パラメータ経由のコマンドインジェクションにより、GitHub ユーザーアクセストークンを窃取できると報告した。
    - branch 値が環境セットアップや git fetch に反映されるため、Codex Web では生成タスクの出力から平文トークンを取得し、GitHub へ横展開できた。
    - 同手法は ChatGPT 上の Codex Web だけでなく、Codex CLI、SDK、IDE 拡張でも再現され、ローカルの auth.json に保存された認証情報も悪用可能だった。
    - さらに GitHub ブランチ名自体に悪性ペイロードを仕込むことで、複数利用者の GitHub トークンを自動で外部サーバーへ送信させる PoC も示された。
    - 悪性ブランチを既定にしたり PR コメントで @codex を呼ぶことで、コードレビュー用コンテナから GitHub Installation Token も奪えるとされた。
- IOCの列挙
    - IOC情報なし
- 推奨事項
    - AI エージェント開発側は、GitHub など外部由来を含む全てのユーザー制御入力をシェルへ渡す前にサニタイズし、文字列補間ではなく安全な API やパラメータ化を使う。
    - コマンドラインに触れる入力ではシェルメタ文字やバッククォートをエスケープまたは遮断し、トークン権限と有効期間を最小化する。
    - 利用組織は GitHub 上の AI アプリ権限を定期監査し、異常なブランチ名や Unicode 空白を監視しつつ、GitHub トークンの定期ローテーションとアクセスログ確認を行う。
- その他
    - なし
- ChatGPTの推奨事項
    - GitHub で Codex など AI アプリに与えた権限を直ちに棚卸しし、不審なブランチ名の有無確認と GitHub トークンのローテーション、アクセスログ点検を実施してください。
